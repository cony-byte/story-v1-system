# 발행성과 운영 규칙 (2026-07-05 확정)

> 자체발행 영상의 성과를 규칙엔진에 흡수해 "judge.py 점수 높은 기획이 실제로 잘 됐나"를 검증하기 위한 규칙.
> 입력 시트(Drive) → CSV 내보내기 → `00_규칙엔진/발행성과/`에 투입 → `engine.py` 재실행이 전체 흐름이다.

## 1. 연결 키 규칙 — 제일 중요

**발행하는 순간 `episode_id`를 기록한다.** 이게 없으면 성과가 돌아와도 어느 기획인지 영영 연결 못 한다.

- 형식: **`{character_id}/{ep_id}`** — 예: `char_seojun/ep_01`
  (ep_01은 캐릭터마다 겹치므로 반드시 캐릭터까지 붙인 복합키를 쓴다. 이 키가 `03_캐릭터/{character_id}/episodes/{ep_id}.json`과 1:1로 매칭되고, 엔진이 이 키로 기획의 target_trigger를 찾아 크롤링 통계에 합산한다.)
- 기록 위치: 발행 로그 시트에 한 줄 (+ 여력 되면 영상 설명 끝에도 코드 삽입)
- **실무자와 지금 합의할 것** — 소급은 지옥이다.

## 2. 성과 CSV 컬럼 (템플릿: `발행성과_템플릿.csv`)

```
episode_id, platform, video_url, publish_date, measured_date, measure_point,
views, likes, comments, shares, saves, duration_s, notes
```

- 지표 필드명은 크롤링 스키마와 동일 (schema.md 통일 원칙) → 엔진이 크롤링 데이터와 같은 데이터 포인트로 흡수
- ER%·저장률%는 입력하지 않는다 — 엔진이 `(saves+shares+comments)/views`, `saves/views`로 계산 (크롤링과 동일 정의)
- `notes`에 "예시"가 포함된 행은 엔진이 자동 스킵 (템플릿 예시 행 보호)

## 3. 측정 시점 고정

- **기본: 발행 후 D+7 측정** (`measure_point=D7`). 측정 시점이 제각각이면 ER 비교가 무의미해진다.
- 여력 되면 **D+30 2차 스냅샷** — 같은 발행 건에 행을 추가한다 (`measure_point=D30`, 시계열 누적 원칙과 동일하게 덮어쓰지 않고 행 추가).
- 엔진은 같은 발행 건(episode_id+platform+publish_date)의 스냅샷 중 **D7만 규칙 집계에 사용**한다. D30 행은 시계열 기록으로 보존된다.

## 4. 30초 입력 절차 (실무자용)

실무자에게 파일 형식·폴더 구조를 요구하지 않는다:

1. Drive의 **발행성과 입력 시트** (컬럼은 이미 다 만들어져 있음)
2. 발행할 때: `episode_id`(복합키)·platform·video_url·publish_date 한 줄 입력
3. D+7에: views/likes/comments/shares/saves 다섯 칸 입력, measure_point에 `D7`
4. 끝. 시트→CSV 내보내기→`발행성과/` 투입→`engine.py` 재실행은 운영자(또는 추후 스크립트)가 한다.

## 엔진 동작 (참고)

- `발행성과/*.csv`는 `source=own_published`로 로드되어 기준선·트리거 통계에 합산되고, 별도로 `own_published_summary`(에피소드별 반응률·저장률·연결 트리거)로도 집계된다 — 예측 vs 실제 검증용.
- 자체발행은 정의상 본편이므로 `--content-type drama_clip` 필터에서 면제된다.
- 검증 완료: 예시 행 스킵 / D7 우선 채택 / ER 자동 계산 / 트리거 합산 (2026-07-05 합성 데이터 테스트).
